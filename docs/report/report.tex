\begin{abstract}
The system implements the complete path from focused web crawling to search-result
presentation. A resumable, host-aware crawler admits English Tübingen pages. A
self-implemented BM25F ranker adds term proximity, and an MPNet-based semantic
score is fused with the lexical score. The interface preserves a ranked list
while mapping results to an interactive constellation. On one frozen crawl and
15 queries, an archived controlled ablation records an increase in
$\mathrm{nDCG}@10$ from 0.599 for the lexical baseline to 0.655 after stopword
filtering and proximity. These judgments were pooled from system output, so the
numbers are diagnostic rather than an unbiased estimate of generalization.
\end{abstract}

\section{Focused Crawling and Storage}
Eight workers share one priority frontier. Each host owns a URL heap. A host
remains leased while its page and any same-origin sitemap are fetched, so the
crawler does not issue concurrent requests to that host. After the lease, the
scheduler waits for the global 0.7-second request delay before starting the next
request to that host.
Every URL is checked against a cached \texttt{robots.txt} policy before fetching
\cite{koster2022robots}. HTTP 429 and 5xx responses receive at most the
configured number of attempts and a delay capped at 30\,s.

LinkVerdict scores canonical link candidates from anchor, target, parent, and
depth features. Depth and host saturation reduce frontier priority. Per-page,
target-host, URL-family, seed, and host budgets limit concentration. At most
three links per page use the lower-threshold exploration lane. After fetching,
PageVerdict and the parsed language admit only English pages above the fixed
index threshold. A 64-bit SimHash over token trigrams rejects text fingerprints
within Hamming distance three \cite{manku2007simhash}.

Accepted HTML is stored by host. Page, rejection, and link-decision metadata is
stored in SQLite. Frontier URLs, seen URLs, and SimHashes are checkpointed as
JSON and reloaded on restart. The working snapshot contains 7,568 accepted
English pages from 396 hosts and 12,452 recorded rejections. Of the 20,020
accepted or rejected fetch outcomes, 2,233 are near-duplicate rejections
(11.2\%). These are database counts, not estimates of topical precision.

\section{Indexing and Ranking}
\subsection{Classical first stage}
The tokenizer lowercases Unicode alphanumeric tokens, drops one-character
tokens, and removes a fixed English stoplist. The inverted index stores body
positions and precomputed BM25F contributions for body, title, and URL-path
fields with weights $1$, $5$, and $3$. With $b=0.75$ and $k_1=1.2$, the
implementation uses
\begin{eqnarray*}
\tilde f_{t,d} &=& \sum_f w_f
\frac{f_{t,d,f}}{1-b+b|d_f|/\overline{|d_f|}},\\
s_{\rm BM25F} &=& \sum_{t\in q}\mathrm{IDF}(t)
\frac{\tilde f_{t,d}}{k_1+\tilde f_{t,d}} .
\end{eqnarray*}
This preserves term-frequency saturation after field weighting
\cite{robertson2004bm25f}. For the distinct query terms matched in the body, the
smallest covering window adds $0.25/(1+g)$, where $g$ is the number of
non-query positions inside that window. A document matching fewer than two
distinct query terms receives no bonus. This is a deliberately weak ordering
signal related to prior proximity extensions of Okapi scoring
\cite{rasolofo2003proximity}.

\subsection{Semantic evidence and fusion}
Offline, each document is split into at most 20 character passages of length
2,000 with 200-character overlap. The title and passages are encoded by
\texttt{nli-mpnet-base-v2} from Sentence Transformers. Attention-mask-aware
mean pooling and $L_2$ normalization produce 768-dimensional vectors. At query
time, cosine similarities form
\[
s_{\rm sem}=0.20s_{\rm title}
 +0.65\max_i s_i
 +0.15\,\mathrm{mean}(\mathrm{top3}_i\,s_i).
\]
Sentence-BERT-style representations support direct cosine comparison
\cite{reimers2019sentencebert}. The union of the top 100 lexical and top 100
semantic documents is re-ranked by
$0.7\hat{s}_{\rm lex}+0.3s_{\rm sem}$, where $\hat{s}_{\rm lex}$ is min--max
normalized over that union. Missing, malformed, model-mismatched, or
index-mismatched vectors fall back to lexical ranking.

\paragraph{Compliance boundary.}
The current union admits semantic-only candidates. It is therefore a hybrid
candidate generator, not a strict re-ranker of only the classical top 100. A
strict reading of the project FAQ requires removing the semantic top-100 branch
and applying $s_{\rm sem}$ only to lexical candidates. Figure~\ref{fig:architecture}
makes this deviation explicit. The crawler, index, search, API, and batch
pipeline are Python. The interactive client is React/TypeScript despite the FAQ
stating that everything must be implemented in Python. The client requests 30
results, although the API and batch path support 100.

\section{Evaluation}
We use graded labels 0--3 and nDCG, which discounts relevant documents at lower
ranks \cite{jarvelin2002ndcg}. Table~\ref{tab:results} reproduces archived JSON
metrics for three runs on the same crawl snapshot and the same 15 queries.
Stopword filtering changed nDCG@10 by $+0.043$. Proximity changed it by a
further $+0.012$. These are observed differences, not significance-tested
effects.

The archived 20-query hybrid diagnostic records nDCG@10 0.712,
nDCG@20 0.694, MRR@10 0.935, 8.55 positives@10, and 55.16\,ms mean measured
retrieval time for 100 results. It is not a controlled estimate of the
semantic component's incremental value. All reported runs were evaluated
against a qrels pool expanded from system outputs, giving 100\% judged coverage
at ranks 10 and 20. No independent assessors, held-out query set, or confidence
intervals are available. Stronger causal claims are therefore not supported.

\section{Evidence and Reproducibility}
Crawl counts above are produced by the crawl-report command directly from the
current SQLite database. The lexical rows in Table~\ref{tab:results} and the
hybrid diagnostic are archived as JSON in the retrieval benchmark's runs
directory. Each record fixes the crawl name, query count, metric values, and
timestamp.

Latency is measured with \texttt{time.perf\_counter} around retrieval for each
query. It includes query encoding in a warmed process, but excludes process
startup, index and vector loading, HTTP transport, and rendering. We therefore
do not label it end-to-end latency. The repository contains runtime classifier
artifacts but no held-out PageVerdict or LinkVerdict metrics. No classifier
accuracy is claimed.

\section{Result Presentation}
The FastAPI service loads the index and matching vectors once at startup and
accepts up to 100 results per request. The batch command defaults to 100 and
writes query ID, rank, URL, and score as tab-separated rows.

The web client presents ten clickable result cards per page together with ten
result stars. Star size and color encode the result-set min--max-normalized final
score. Without category labels, vertical position follows rank. Users may name
an $x$ and/or $y$ category. The server embeds each label and projects mean
document vectors onto it, after centering and bounded scaling. Overlapping stars
fan out on hover, and pagination pans the camera to the next ten-result
constellation. The ranked cards remain visible, so the spatial view augments
rather than replaces direct navigation.

\section{Conclusion}
The system supplies a focused crawl, local index, batch path, API, and innovative
interface. Its strongest controlled evidence is limited to the
lexical ablation: stopword filtering plus proximity raised nDCG@10 from 0.599
to 0.655 on one fixed 15-query setup. Independent judgments on a frozen crawl
and a strictly lexical candidate set are still required to establish the
incremental value of semantic fusion and full compliance with the brief.
